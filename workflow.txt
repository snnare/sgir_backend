==============================================================================
SGIR - FLUJOS DE TRABAJO (WORKFLOWS) ACTUALIZADOS
==============================================================================

Este documento describe la interacción lógica entre los servicios y endpoints,
reflejando la arquitectura modular y el monitoreo dinámico implementado.

------------------------------------------------------------------------------
1. FLUJO DE AUTENTICACIÓN Y SEGURIDAD
------------------------------------------------------------------------------
1. Registro: POST /users/ -> Service (hashing.py) genera hash Bcrypt.
2. Login: POST /users/login -> Service (hashing.py) verifica contra DB -> Service (tokens.py) emite JWT.
3. Autorización: Dependencia get_current_user valida el token en cada petición protegida.

------------------------------------------------------------------------------
2. FLUJO DE GESTIÓN DE ACTIVOS (INVENTARIO)
------------------------------------------------------------------------------
1. Servidor: POST /servidores/ -> Registro de IP y criticidad.
2. Credenciales: POST /credenciales/ -> Service (encryption.py) cifra la contraseña con Fernet.
   *Nota: Esto permite que el backend use la contraseña para monitoreo pero no la exponga en texto plano en la DB.*

------------------------------------------------------------------------------
3. FLUJO DE MONITOREO DINÁMICO (AD-HOC)
------------------------------------------------------------------------------
Propósito: Obtener el estado actual de un motor sin configurar scrapers fijos.

1. Petición: GET /monitoring/{engine}/{server_id}/{cred_id}
2. Backend:
   a. Recupera servidor y credencial desde app/services/infrastructure_crud.py.
   b. Desencripta contraseña (encryption.py).
   c. Dynamic Core genera conexión temporal (Engine SQL o MongoClient).
3. Service:
   - MySQL (5/8): Ejecuta SHOW GLOBAL STATUS.
   - MongoDB: Ejecuta db.serverStatus().
4. Resultado: Devuelve métricas estandarizadas (Uptime, Conexiones, QPS, Memoria).

------------------------------------------------------------------------------
4. FLUJO DE AUDITORÍA INTEGRADA (COMPLIANCE)
------------------------------------------------------------------------------
Cada acción exitosa en los módulos de 'core_crud' o 'monitoring' dispara una
llamada al servicio audit_crud.log_event().

Registro automático:
- Quién: id_usuario extraído del token JWT.
- Qué: Entidad afectada (Servidor, Usuario, Política, etc.).
- Resultado: Eventos de Creación, Modificación, Eliminación o Ejecución.

------------------------------------------------------------------------------
5. FLUJO DE PROTECCIÓN DE DATOS (BACKUPS)
------------------------------------------------------------------------------
1. Configuración: Definir Rutas y Políticas de Respaldo.
2. Vinculación: Asignar política a una Base de Datos específica.
3. Trazabilidad: Registrar ejecuciones manuales o automáticas en el historial de respaldos.
